# AI運用ルール（Project Guardrails）

## AIの必読フロー（作業開始時）
1. `00_START_HERE.md` を読む
2. `docs/03_status.md` を読む（今動くもの / 次の3つ）
3. 不明点は仮定せず質問し、作業計画（箇条書き）を提示してから実装する

## このリポジトリの前提
- **ファイルを読む/書くのはAIのみ**（人間はIssue/PRで自然言語の指示と承認を行う）
- 作業はPR経由で進め、`main` を壊さないことを最優先にする

このリポジトリは、非エンジニアが自然言語で指示し、AIが実装を先導する前提で作られています。
最優先は「事故防止」と「中断しても再開できる状態」です。

## 役割
- AI: 実装ドライバー。計画、実装、確認、ドキュメント更新、Pull Request作成を行う。
- 人間: プロダクトオーナー。要件の確定と承認、レビュー判断を行う。

## 絶対ルール
1. 不明点は仮定しない。質問してから進める。
2. 変更前に作業計画を箇条書きで提示し、承認後に実装する。
3. Bootstrap Issue（起動）では、最初のPRは台帳（docs）を確定させるだけにする（コード実装はしない）。
   - 対象: docs/00_goal.md, docs/01_requirements.md, docs/02_runbook.md, docs/03_status.md
4. 変更はPull Requestで行う。1PRは小さくし、無関係な変更を混ぜない。
5. 機密情報を出さない。コード、コミット、Issue/PR本文、ログ貼り付けに書かない。`.env` をコミットしない。
6. `main` を壊さない。自動チェックがある場合は必ず通す。ない場合も最小の確認手順を残す。
7. 実装後に必ず更新する。
   - docs/02_runbook.md（実行手順）
   - docs/03_status.md（現状と次の一手）
   - 必要に応じて docs/04_decisions.md（判断ログ）, docs/05_ai_log.md（作業ログ）
8. 中断する場合は `prompts/40_stop.md` に従い、再開可能な状態にして止める。
9. ファイル削除、大きな構造変更、GitHub Actionsの変更は事前に人間へ確認する。

## 作業計画（Implementation Plan）の書き方
- 目的: Issueの受け入れ条件にどう対応するか
- 変更箇所: 触るファイルやディレクトリ
- PR分割案: どの順に、何PRに分けるか
- 確認方法: 実行コマンドと期待結果
- ロールバック案: 取り消し方法

## Definition of Done
- 受け入れ条件を満たす
- 確認手順が再現できる
- docsが更新されている（少なくとも runbook と status）
- 機密情報が含まれていない
- PRテンプレのチェックリストが埋まっている
