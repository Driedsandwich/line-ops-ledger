# 設計メモ（Design）

このファイルは「あとから保守できる」ための高レベル設計メモです。
最初から完璧に書く必要はありません。重要な判断と前提だけ残してください。

## システム全体像
- 画面（UI）:
  - React SPA。主要ルートは `/`（ダッシュボード）/`/lines`（主台帳）/`/lines/history`（契約履歴）/`/settings/*`（設定）。
- バックエンド:
  - 外部バックエンド/APIなし。保存と表示はブラウザ側で完結。
- データ保存:
  - `localStorage` を主データストアとして利用。`src/lib/storage.ts` で永続化状態を確認し、必要なら `StorageManager` 経由の状態監視を行う。
- 外部API:
  - 基本運用では通知トリガー・解析・同期のための外部 API 依存を前提しない。
- 認証（必要なら）:
  - 現在は未実装。将来、端末横断同期や共有要件が出た場合に再設計する。

## データモデル（概念でOK）
- エンティティ1:
  - 主台帳（LineDraft）:
    - 回線識別情報、契約状態、費用、期限系データ、利用実績、特典情報
- エンティティ2:
  - 履歴（LineHistory / 活動ログ）:
    - 電話番号単位の履歴と、履歴内の複数活動ログ、活動種別、活動日、メモ

## 外部依存
- 使うサービス:
  - なし（アプリ本体のデータ保存・UI動作は外部サービス非依存）
- キーの置き場所:
  - `.env`（Gitに入れない）
  - 開発ラベルは `.env.local`（Git ignore）で運用

## 主要フロー
- サイドパネル標準: アプリ全体の主要遷移は左側ナビ（サイドパネル）で統一し、見出し付きセクション（メイン / 履歴 / 設定）＋各リンクで構成する。
  - ナビは常時利用でき、主要ルート（`/`, `/lines`, `/lines/history`, `/settings/*`）の入口を明確に提示する。
  - ルート到達性を優先し、重複・混在リンクは同一セクション内に正規化する。

- フローA:
  - `/lines` で台帳を編集し、必要に応じて `/settings/storage` と `/settings/backup` の導線へ遷移して保全運用
- フローB:
  - `/lines/history` で履歴を追加し、`quickActivity`/`historyIntent` 文脈で台帳や入力画面へ往復
  - ダッシュボードからの要アクション導線で主台帳・履歴へ自然遷移

## テスト方針（要約）
- 自動で担保するもの:
  - `npm run check`（型チェック）と `npm run build`（ビルド）
  - CI `check-and-build`（`npm ci` -> `npm run check` -> `npm run build`）
- 手動で確認するもの:
  - 主要画面(`/`, `/lines`, `/lines/history`, `/settings/*`)の表示と主要導線
  - 空状態ガイド、履歴保存、バックアップ入出力、主要なドリルダウン条件
