# 目的（Goal）

## 1. プロダクト名
- `line-ops-ledger`

## 2. 一文で言うと何を作るか
- 複数回線の契約・履歴・期限・特典を、`line-centric / local-first` で扱う command center 型の管理ツール。

## 3. 何を解決するか（Problem）
- いま困っていること:
  - 回線ごとの契約開始日・解約可能時期・MNP 期限・特典期限・活動履歴が別々になりやすい
  - 失効しやすい期限と、記録すべき活動が埋もれやすい
  - 複数回線を少ない画面遷移で確認・更新したい
- 放置した場合の損失:
  - 特典失効、期限見落とし、長期未活動、履歴の記録漏れ

## 4. 誰のためのツールか（Target user）
- 利用者:
  - 複数回線を扱う運用担当者
  - 自分で回線管理をしたい非エンジニアのプロダクトオーナー
- 利用シーン:
  - 日次で dashboard を見て危険案件を処理する
  - `/lines` で回線詳細を更新する
  - `/lines/history` で活動を記録する
  - `/settings/backup` で local backup を入出力する

## 5. 成功条件（測定可能）
- 指標1:
  - `/` で危険案件・要アクション・健全性を 1 画面で確認できる
- 指標2:
  - `/lines` と `/lines/history` の drilldown から、必要な回線・履歴・期限へ迷わず到達できる

## 6. 非目的（やらないこと）
- `devices / tasks / secret vault / calendar / ROI` の大規模な別ドメイン化
- 外部バックエンドへの依存
- 機密情報の新規収集・平文の機密 vault 化
- 収支予測タイムラインや系列ペナルティの先行実装

## 7. 制約（Constraints）
- 予算（従量課金の上限など）:
  - 追加の外部 SaaS コストは前提にしない
- 期限:
  - 固定期限なし
- 実行環境（Windows/Mac、社内PC制限など）:
  - Windows 上のローカル開発
  - ブラウザ実行前提
- 取り扱いデータ（個人情報、社内秘など）:
  - 電話番号・契約者名義・履歴・特典条件などの個人関連データ
  - 外部送信しない前提
- 使えないもの（外部API禁止など）:
  - 外部バックエンド必須の設計
  - 機密情報を前提にした remote vault

## 8. 承認が必要な人（Stakeholders）
- プロダクトオーナー

## 9. 既知のリスク
- localStorage / browser cache の破損・古い SW による表示崩れ
- 仕様を広げすぎることによるドメイン肥大化
- 既存 drilldown や URL パラメータの回帰
