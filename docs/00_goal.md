# 目的（Goal）

## 1. プロダクト名
- line-ops-ledger（回線運用台帳）

## 2. 一文で言うと何を作るか
- 個人運用の通信契約台帳を local-first で管理し、回線・履歴・期限・特典・光回線残債・次アクションを command center で確認できる Web アプリ

## 3. 何を解決するか（Problem）
- いま困っていること:
  - 契約開始日、解約可能日、MNP 期限、無料オプション期限、特典期限、次回確認日が分散しやすい
  - 履歴の記録と回線一覧の確認が別画面に分かれており、必要な操作への導線が切れやすい
  - 目視だけでは、長期未活動、利用実績不足、光回線残債、名義別の偏りを見落としやすい
  - ローカル保存だけで運用するため、バックアップや復元の確認を怠ると復旧できなくなる
- 放置した場合の損失:
  - CB 失効、期限超過、解約タイミングの逸失、確認漏れ、復元不能、記録の重複・欠落

## 4. 誰のためのツールか（Target user）
- 利用者:
  - 現時点では単独運用の個人
  - 将来的には家族や小規模チーム利用も想定するが、現段階の主対象ではない
- 利用シーン:
  - 回線を登録・編集する
  - 履歴と活動ログを記録する
  - 期限・特典・残債・利用実績を確認する
  - バックアップを取り、必要時に復元する

## 5. 成功条件（測定可能）
- ダッシュボードで、今やるべきことを 3 層で把握できる
- `openDraft` / `focusSection` / `contractActiveOnly` / `usagePriority` の drilldown が壊れない
- 履歴入力の quick logging、下書き復元、候補管理、イベント一覧が再読み込み後も維持される
- JSON バックアップのエクスポート / インポートで主要データが失われない
- 重要な秘匿情報を平文 localStorage に残さない方針を維持する

## 6. 非目的（やらないこと）
- キャリア API の自動連携
- 複数ユーザー権限管理
- 会計ソフト連携
- 高度な税務計算
- スクレイピング
- 統合カレンダー / 収支予測タイムライン / 系列ペナルティマトリクスの先行実装
- devices / tasks / secret vault / carrier credentials の巨大な再設計を今すぐ本体へ持ち込むこと

## 7. 制約（Constraints）
- 予算（従量課金の上限など）:
  - 当面は local-first 前提で追加コストなし
- 期限:
  - 明示的な最終期限は未確定
- 実行環境（Windows/Mac、社内PC制限など）:
  - CodexDesktop / Windows / PowerShell 前提
- 取り扱いデータ（個人情報、社内秘など）:
  - 電話番号、契約情報、活動履歴、特典情報、将来的には秘匿情報を含む可能性がある
- 使えないもの（外部API禁止など）:
  - 現フェーズでは外部 API 連携を前提にしない

## 8. 承認が必要な人（Stakeholders）
- プロダクトオーナー
- レビュー担当
- 将来の自分自身（運用継続者）

## 9. 既知のリスク
- スコープが devices / tasks / calendar / ROI / secret vault へ広がりすぎる
- 既存の localStorage データと将来のスキーマ差分が大きくなる
- 画面の情報量が増えすぎて command center の見通しが悪化する
- バックアップ / import の検証が弱いまま機能が増える
- 秘匿情報の取り扱い方針が曖昧なまま拡張する
